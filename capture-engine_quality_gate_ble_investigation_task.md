# 作業指示書: capture-engine — 品質ゲート調査（BLE が弾かれる機序の解明）

## 役割
あなたは sigscan プロジェクトの capture-engine エージェント。
本指示は **調査のみ（read-only・コード変更ゼロ）**。目的は、実機収集で **BLE（狭帯域・間欠バースト）が
品質ゲートで捨てられる**現象について、**どのゲートが・なぜ・どの条件で弾くのかをコードから解明**し、
「BLE は通すがスプリアス/ノイズは弾く」調整が可能かを判断する材料を、根拠つきで報告すること。
**修正はしない**（ゲートを緩めるとスプリアス/ノイズも通すため、調整方針は機序を理解した人間が承認して
から別タスク。quality は凍結継ぎ目でもある）。

## 背景（今日の実機実験で判明した事実 — これが最大の手がかり）
2.4GHz 帯の実機収集で、BLE が**3種類の drop 理由**で捨てられることを観測した:

1. **`drop:dc-spike`**: 2400MHz 付近の BLE（SNR 45dB もある明確な信号）が DC スパイクと誤認され破棄。
   `--q-dc-excess` を極端(100)に緩めると保存されるが、今度は DC 線/スプリアスが写り込み確定困難。
2. **`drop:narrow-steady-spur,dc-spike`**: DC 除去を切って収集したところ、
   **2408 / 2424 / 2440 / 2456 / 2472 / 2488MHz にきっちり 16MHz 間隔**で、
   **全て BW=0.5MHz・SNR 41-44dB・persist=1.00（全時間連続）**の信号が並び、全て
   `narrow-steady-spur`（細い定常スプリアス）として破棄された。
   → **この等間隔・同幅・同強度・常時オンの櫛は、HackRF 内部が生むスプリアス**（本物の電波ではない）。
   ゲートは正しくこれを弾いている。**ただし、細い BLE も同じ「narrow」条件に巻き込まれて
   一緒に弾かれている疑い**がある（BLE も 0.5-1.5MHz と細いため）。
3. **`drop:low-persistence(0.26<0.34)`**: 範囲を狭めて BLE を狙うと、
   `persist=0.26(5/19)`（19観測中5回だけ信号あり）で「持続性が低い」として破棄。
   → **BLE は間欠（ホッピングでたまにその周波数に来る）ため、1周波数を10秒見ても
   一部の観測にしか現れず、持続性ゲート（閾 0.34）に引っかかる**。

**核心的な問い**: BLE（狭帯域・間欠・バースト）と、HackRF スプリアス（狭帯域・**定常**・常時オン）は、
「細さ」は似るが「時間構造（間欠 vs 定常）」が違う。**この違いでゲートが両者を区別できているか、
それとも BLE も一緒に弾いているか**を、コードと実データで確定する。

## 最重要原則（絶対厳守）
1. **調査のみ・コード変更ゼロ**: quality.py / scheduler.py / dsp.py / spec.py / sigmf_io.py など
   一切編集しない。診断用の一時ファイルは captures/ 外・非コミット（scratchpad 等）。
2. **凍結継ぎ目の認識**: `quality.evaluate`（および関連）は6継ぎ目。今回は調査のみだが、
   調整案を書く際も「返り値の形を保ったまま閾値/条件をどう変えるか」の枠内で提案（仕様を壊す案は出さない）。
3. **読み取り専用**: captures/ の `*.sigmf-*` を書き換えない。メタを読むなら sigmf_io の
   read 経路（locale=cp932）。UTF-8 決め打ち禁止。
4. **推測で断定しない**: 「たぶん」で片付けず、**実際のコード行と、実データ/実測値**で裏を取る。
   仮説（BLE がスプリアスと同じ narrow 条件で巻き込まれている等）が外れていたら正直に報告。

## 調査内容（実施して報告に含める）

### (A) 品質ゲートの全体像
1. `quality.py`（および `scheduler.py` の品質評価呼び出し）を読み、**BLE が引っかかった3つの drop**を
   出している箇所を特定して報告:
   - **dc-spike 判定**: どの条件で「DC スパイク」と判定するか（`--q-dc-excess` / `--q-dc-std` が
     効く箇所。中心ビンの突出 dB・観測間の時間不変性の閾）。2400MHz の BLE が誤認される理由。
   - **narrow-steady-spur 判定**: 「細い・定常のスプリアス」と判定する条件（帯域幅の閾・
     「steady=定常」をどう測るか。観測間の変動・持続性で定常性を見ているか）。
     **BLE のような「細いが間欠」の信号が、この条件に該当してしまうか**を条件式から判断。
   - **low-persistence 判定**: 持続性 persist の計算方法と閾（0.34）。19観測中何回で persist=1.0 か。
     BLE の間欠性がなぜ 0.26 のような低値になるか。
2. これらの閾値の既定値と、CLI で調整可能なもの（`--q-*`）を一覧。

### (B) 実データでの裏取り
3. 今日の収集データ（captures/ と、可能なら drop されたログに対応する survey 情報）を使い、
   **BLE と スプリアスの時間構造の違いが数値に出るか**を確認して報告:
   - HackRF スプリアス（16MHz 間隔・persist=1.00・定常）と、BLE（persist が低い・間欠）の
     **persist 値・観測間変動の実測差**。両者が persist で分離できるか。
   - narrow-steady-spur 判定が「帯域の細さ」だけで弾いているのか、「定常性」も条件に入れているのか。
     もし細さだけなら、BLE（細いが間欠）も巻き込む → 「定常性」を条件に加えれば分離できる可能性、を検討。
   - ※実データが該当ケースを含まない場合は、読める記録で該当項目を確認し、その旨を明記
     （捏造しない。読めた範囲で）。
4. **dc-spike 判定**について: 2400MHz 付近の BLE がなぜ誤認されるか。中心ビン突出が
   「DC 位置に BLE が重なった」ために起きるのか、を条件式と（あれば）実データで確認。

### (C) 機序の結論と、区別可能性の判定
5. 3つの drop それぞれについて、**BLE が弾かれる理由**をコード該当行を引用して結論づける。
6. **最重要**: 「BLE（狭帯域・間欠）」と「HackRF スプリアス（狭帯域・定常）」を、
   **品質ゲートが区別できているか／できていないか**を判定:
   - 区別できている（例: persist や定常性で分離）なら、どのゲートがどう分けているか。
   - 区別できていない（BLE もスプリアスと同条件で弾く）なら、**どの条件が過剰か**を特定。
7. dc-spike / low-persistence については、「BLE を捉えるにはどの前提が障害か」を明確化
   （例: 「1周波数滞在」型の持続性評価が間欠 BLE と根本的に相性が悪い、等）。

### (D) 調整方針の提案（提案のみ・実装しない）
8. 「BLE は通すが スプリアス/ノイズは弾く」ための**調整候補**を、案として文章で書く。凍結シグネチャを
   保つ枠内で。例（実データの機序に合わせて具体化）:
   - narrow-steady-spur に「定常性」条件を効かせ、間欠 BLE を除外対象から外す。
   - low-persistence 閾（0.34）を下げる or 間欠信号向けの別評価を検討（ただしノイズ誤通過の trade-off）。
   - dc-spike 判定を範囲設定（BLE を DC 位置から外す）で回避 vs 閾調整で回避、の比較。
   **各案の trade-off（スプリアス/ノイズを通すリスク）を必ず明記。実装は一切しない。**
9. 各案が影響する範囲（既存テスト test_quality 等、他の drop 挙動、scheduler 連携）を洗い出す。
