# 作業指示書: バンドプラン拡張（1〜6GHz「完全な地図」化）

## 役割
あなたは sigscan プロジェクトの capture-engine エージェント。
本指示は `config.py` の `BAND_PLAN` を、総務省一次情報に基づく追加提案
（別添 `BANDPLAN_PROPOSAL.md` の §3 / §5）に沿って拡張する作業である。

## 最重要原則（絶対厳守）
1. **凍結契約に触れない**: `spec.py` と `sigmf_io.py` は一切変更しない。
   6つの継ぎ目（spec.render / sdr backend / dsp.measure_signal,detect_segments /
   classify.classify / sigmf_io read+write / store.Store）のシグネチャを変えない。
2. **「想定用途/方式」は初期仮説**: 提案書 §3 の想定用途・想定方式は、一次情報に
   基づく **初期仮説（地図上の予測）** である。**決して「確定ラベル」として
   分類器に決め打ちさせない**。実測データの確信度でラベルが確定する設計を壊さない。
   - 具体的には: バンドに用途/方式のヒントを持たせてよいが、それを classify が
     「確信度を無視して機械的に用途を割り当てる」ことに使ってはならない。
   - 既存のラベル確定フロー（確信度<0.7 → Unknown、review.py で人間判断）は不変。
3. **データ汚染を起こさない**: 既存の収集済みデータ（captures/）や保存形式に
   影響を与えない。BAND_PLAN の追加は「今後どこをスキャンするか」の地図であり、
   過去データの再解釈やラベル上書きを行わない。
4. **Windows/cp932 制約**: 出力は UTF-8 強制の既存方針を踏襲。新規にファイルを
   開く場合の encoding も既存コードの流儀に合わせる（.sigmf-meta は locale/cp932、
   その他は UTF-8 強制など、既存に倣う）。日本語コメントは既存同様に可。

## 事前確認（実装前に必ず行う）
1. `config.py` を読み、`Band` dataclass の正確なフィールドを確認する
   （既知: name, f_lo, f_hi, priority, hint。f_lo/f_hi は Hz 単位の float）。
2. 現在の `BAND_PLAN` の全エントリを把握する（重複追加を避けるため）。
3. `BAND_PLAN` を参照しているコード（scheduler.py の survey/巡回、classify.py、
   main.py など）を grep し、エントリ追加が既存ロジックを壊さないか確認する。
   - 特に、巡回順序・priority の使われ方・f_lo/f_hi の前提（昇順か等）を確認。
4. 既存テストを確認し（tests/）、BAND_PLAN に関する前提（件数固定など）が
   テストにあれば、それも更新対象に含める。

## 実装内容

### (1) 既存エントリの修正
- **R3**: 現行「Aero/Radar S-band」(2.700-2.900GHz) の上限を **3.400GHz** に拡張。
  一次情報「2700-3400 船舶の航行用等レーダー」に合わせる。
  - name も実態に合わせて調整可（例: "Sバンド航行用レーダー" 等）。ただし既存 name を
    参照しているコード/テストがあれば壊さないこと（grep で確認のうえ判断）。

### (3) 想定用途/方式マッピングの持たせ方（提案書 §5-2）
- 上表では hint 文字列に `[仮説] use=... mod=...` を埋め込む方式を採用している。
  これが既存 hint の使われ方と整合するか確認すること。
- **もし** hint が表示専用で機械可読でないなら、別途
  `BAND_LABEL_HINT: dict[str, dict]`（バンド name → {"use_hint":..., "mod_hint":...,
  "confidence_cap": ...}）のような **任意の参照表** を追加する方が綺麗。
  - この表は **classify が「ヒント」として参照するだけ** に留め、確信度を無視した
    決め打ちには使わない（原則2）。
  - 実装するかは、既存 classify の構造を見て、無理なく入る方を選んでよい。
    入れない場合は hint 文字列のみとし、機械可読マッピングは将来課題とする
    （その旨をコメントか FILES.md に記録）。
- **判断に迷ったら**: 最小実装（hint 文字列のみ、BAND_PLAN 拡張だけ）を優先。
  X注記やラベル対応表の機械可読化は「将来課題」としてコメントに残す。

### (4) X注記（混入・誤認要因, 提案書 §X）の扱い
- X1〜X4（2.4GHz WPT, 5GHz気象レーダ同居, 2.4GHzアマチュア重複 等）は
  **今回はコード上の注記（コメント）として残すに留めてよい**。
  quality.py / classify.py の該当箇所に「# [混入注意] 2412/2437/2462/2484MHz は
  WPT が WiFi と誤認されうる（BANDPLAN_PROPOSAL §X1）」のようなコメントを置く。
- 実際の弾き処理（WPT判定ロジック等）は **将来課題**。今回は実装しない。
  （原則: 一歩ずつ。まず地図、ロジックは後）

## 検証（実装後に必ず行う）
1. `python -m py_compile *.py` が通る。
2. `python -m pytest -q` で全テストが通る（既存件数 + 変更分）。
   - BAND_PLAN 件数に依存するテストがあれば適切に更新し、なぜ変えたかコメント。
3. 凍結契約の差分ゼロを確認: `git diff --stat -- spec.py sigmf_io.py` が空。
4. BAND_PLAN が正しく拡張されたことを表示で確認:
   `python -c "import config; [print(f'{b.f_lo/1e9:.3f}-{b.f_hi/1e9:.3f}GHz prio={b.priority} {b.name}') for b in config.BAND_PLAN]"`
   - 周波数順に並んでいなくても良いが、追加分が全て存在することを確認。
5. scheduler の巡回が新バンド追加で異常終了しないこと（短時間の SimBackend 巡回 or
   ドライランで確認できれば尚良い。実機は不要）。

## 完了報告に含めること
- 変更したファイル一覧と、各変更の要点。
- BAND_PLAN の最終エントリ数（変更前→変更後）。
- 想定用途/方式マッピングを「hint文字列のみ」「参照表追加」のどちらで実装したか、
  およびその理由。
- X注記をどこにコメントとして残したか。
- テスト結果（N passed）、py_compile 結果、凍結契約 diff が空であること。
- 「想定方式は初期仮説であり、確信度による確定フローは不変」を守ったことの明言。
- 将来課題として残した項目（ラベル対応表の機械可読化、WPT等の弾きロジック 等）。

## やってはいけないこと（禁止事項の再掲）
- spec.py / sigmf_io.py の変更。
- 確信度を無視した「周波数→用途」の決め打ち実装。
- 既存の収集済みデータやラベルの上書き・再解釈。
- 提案書にない独自バンドの大量追加（提案 §3 の範囲に限定。疑義は報告して判断を仰ぐ）。
- 重複/隣接を理由にした提案バンドの無断削除。
