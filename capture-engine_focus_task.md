# 作業指示書: 帯域集中機能（--focus）の実装

## 役割
あなたは sigscan プロジェクトの capture-engine エージェント。
本指示は「指定した周波数範囲に張り付き、バンドプラン巡回による範囲外への
寄り道をやめる」機能（帯域フォーカス）を実装する作業である。

## 背景（なぜ作るか）
- BAND_PLAN が 31 バンドに拡張された（1〜6GHz 完全地図化）。
- 現状、`--start/--stop` を指定してもサーベイ範囲が絞られるだけで、
  HybridScheduler のバンドプラン巡回は **全バンド** を回り続ける。
  実測ログでは各サイクルの過半が範囲外バンドへの滞在（persist=0.00 の空振り多数）。
- 「2.4GHz 専門家」のデータ収集では、単一 HackRF の時分割リソースを
  指定帯域に全振りしたい。これが本機能の目的。

## 最重要原則（絶対厳守）
1. **凍結契約に触れない**: `spec.py` / `sigmf_io.py` は一切変更しない。
   6つの継ぎ目（spec.render / sdr backend (sweep_power, capture_iq) /
   dsp.measure_signal, detect_segments / classify.classify /
   sigmf_io read+write / store.Store）の **シグネチャを変えない**。
   - 特に `sweep_power` のサーベイ範囲ロジックは触らない（既に start/stop で
     絞られている。問題はそこではない）。
2. **既定は OFF（挙動不変）**: 新フラグ `--focus` が無いときの動作は、
   現在と 100% 同一であること。既存テストは1件も変更せずに通ること
   （フォーカス用の新規テスト追加は可）。
3. **関所は1点に集約**: 範囲外除外のロジックは、ターゲット候補の合流点
   1箇所（_build_targets 相当の出口）にのみ置く。survey / dwell / classify /
   保存系に if を散らさない。
4. **データ来歴の記録**: フォーカス収集されたデータには SigMF グローバルに
   `sigscan:band_focus` を記録する（`sigscan:dc_removed` と同じ流儀）。
   sigmf_io の write シグネチャは不変のまま、呼び出し側（scheduler）が
   グローバル dict に key を足す形で実現すること（既存の dc_removed /
   capture_mode の記録方法を踏襲）。

## 事前確認（実装前に必ず行い、結果を報告に含める）
1. `scheduler.py` を読み、ターゲット候補が合流する正確な箇所
   （_build_targets またはそれに相当する処理）と、バンドプラン由来 /
   サーベイ由来の候補がどう混ざるかを特定する。
2. **ノイズ床推定の局所性を確認する**: ノイズ床（noise floor）の計算が
   サーベイ／各キャプチャの局所データで完結しており、「全帯域を巡回すること」に
   依存していないことをコードで確認する。
   - もし依存が見つかった場合は **実装を中断し、依存の内容を報告して判断を仰ぐ**
     （勝手に回避策を実装しない）。
3. `config.py` の Config/SDRConfig 等の構造を確認し、フラグの置き場所
   （既存の dc_removal などと同じ流儀になる場所）を決める。
4. `main.py` の CLI 引数定義を確認し、`--focus` の追加位置と、
   start/stop が未指定で `--focus` だけ与えられた場合の扱いを決める
   （推奨: start/stop 無しの --focus はエラーまたは警告つき無視。
   どちらにしたか報告すること）。
5. 既存テスト（tests/）の scheduler 関連を確認し、
   「フォーカス OFF の従来挙動」がどのテストで担保されているか把握する。

## 実装内容

### (1) config: フラグ追加
- フォーカス有効/無効のフラグ（既定 False）を Config に追加。
  範囲は既存の start/stop（スキャン範囲）を流用し、新たな範囲指定は作らない。

### (2) scheduler: 関所の実装
- ターゲット候補の合流点の **出口** に1点だけフィルタを置く:
  「focus 有効 かつ 候補の中心周波数が [start, stop] の外 → 候補から除外」。
- これにより以下の両方が同じ関所で消えることを意図している:
  a. バンドプラン巡回由来の範囲外目標（例: 2.4GHz 指定時の GPS 1176MHz, W56 5597MHz）
  b. サーベイ端の食み出し検出（例: --stop 2.5e9 での 2504/2512MHz）
- 注意: 端の食み出し（b）の **根本原因修正は本タスクの範囲外**（別件）。
  関所による除外はフォーカス時のみの実害止血であり、フォーカス OFF 時の
  食み出し挙動は現状のまま残ってよい。

### (3) SigMF 来歴記録
- フォーカス有効で収集された記録の SigMF グローバルに
  `sigscan:band_focus` を追加する。値は最低限 true（bool）でよいが、
  範囲も残せるなら `{"start": ..., "stop": ...}` 等の素直な形でよい
  （どちらにしたか報告）。
- 既存の `sigscan:dc_removed` / `sigscan:capture_mode` の書き方を踏襲し、
  sigmf_io のシグネチャは不変のまま呼び出し側で dict に足すこと。

### (4) main.py: CLI
- `--focus` フラグを追加（store_true）。ヘルプ文は日本語で簡潔に
  （例: "指定した --start/--stop の範囲だけに張り付き、バンドプラン巡回を行わない"）。
- 起動時の表示（mode=... の行の近く）に、フォーカス有効時はその旨を出す
  （例: "帯域フォーカス: 2.40-2.50GHz に集中（バンドプラン巡回オフ）"）。

## テスト（Sim-first・必須4ケース）
SimBackend で以下を新規テストとして追加する（既存テストの変更は不可）:

- **T1 フォーカスONで範囲外バンドプラン目標が消える**:
  focus 有効・範囲 2.4-2.5GHz で targets を構築し、範囲外バンド
  （GPS/W56 等）由来の目標が含まれないこと。
- **T2 範囲外のサーベイ検出も同じ関所で消える**:
  範囲端の外（2504MHz 相当）にサーベイ由来の候補を注入し、
  focus 有効時に除外されること。
- **T3 フォーカスOFFで従来挙動（回帰ロック）**:
  focus 無効時、バンドプラン巡回由来の範囲外目標が **従来どおり含まれる** こと
  （既存挙動が変わっていないことの錠前）。
- **T4 SigMF 来歴**: focus 有効で収集した記録の global に
  `sigscan:band_focus` が存在すること（SimBackend + 一時ディレクトリで）。

## 検証（実装後に必ず行う）
1. `python -m py_compile` 対象ファイル全部 → OK。
2. `python -m pytest -q` → 既存 99 passed を維持しつつ、新規テスト分が増えて
   全部緑（expected: 103 前後 passed, 3 skipped）。
3. 凍結契約 diff ゼロ: `git diff --stat -- spec.py sigmf_io.py` が空。
4. Sim ドライラン（フォーカスON）: SimBackend で
   `--start 2.4e9 --stop 2.5e9 --focus` 相当を短時間実行し、
   `[band:...]` の範囲外滞在ログが出ないことを確認。
5. Sim ドライラン（フォーカスOFF）: 同条件でフラグ無し実行し、
   従来どおり `[band:...]` 巡回が出ることを確認（挙動不変の目視確認）。

## やってはいけないこと（禁止事項の再掲）
- spec.py / sigmf_io.py の変更。6継ぎ目のシグネチャ変更。
- sweep_power / サーベイ範囲ロジックへの変更。
- survey_interval / ノイズ床推定ロジックの変更
  （局所性の確認のみ。依存が見つかったら中断して報告）。
- 既存テストの変更・削除。
- 関所ロジックの分散（複数箇所への if 散布）。
- --start/--stop の暗黙挙動変更（--focus 無しの動作は完全に従来どおり）。
- バンド名指定（--focus-band）等の機能拡張（将来課題。今回は範囲指定のみ）。
