# 作業指示書: cnn-training M1.5 — 実データ推論プローブ

## 役割
あなたは sigscan プロジェクトの cnn-training エージェント。
本指示は、M1 で火入れした Sim 学習済み CNN を使い、実機収集データ
（captures/ の 49 件）に対して **推論のみ** を行い、ドメインギャップを
初観測する「プローブ（探針）」作業である。学習は一切しない。

## 背景（このプローブの三重の価値）
M1 は成功した（Sim val 91.7%、経路 end-to-end 動作）。次に知るべきは
「Sim で覚えた目が、本物をどう見るか」。このプローブは同時に3つを観測する:
1. **ドメインギャップ**: Sim の5クラスの見え方が、実信号にどれだけ通用するか。
2. **13ms くじ引き問題の定量化**: 保存IQは約13msの切り取りのため、まばらな
   バースト（BLE adv 等）はバーストが写っていない記録がありうる。
   「ラベル=BLE なのに 予測=noise-only」の件数は、この問題の**直接の証拠**になる
   （画像にバーストが写っていないなら、noise-only と答えるのは画像の内容としては
   正しい——CNN の故障ではない）。
3. **ルールラベルの監査**: 高確信度で期待と食い違う予測は、ルールラベル側の
   誤りの候補（例: 2480MHz 超の「BLE」ラベル問題）を炙り出す。

## 最重要原則（絶対厳守）
1. **推論のみ。実データで学習・fine-tune しない**（「ルール分類器の出力を
   教師にしない」原則の継続。実ラベルは照合の参考にのみ使う）。
2. **凍結契約不可侵**: spec.py / sigmf_io.py 変更禁止。CNN への入力は
   必ず spec.render 経由（既存の infer.classify_iq の経路を使う）。
3. **軸の違いを混同しない**: CNN のクラスは**方式軸**（wideband-ofdm 等）、
   実データのラベルは**用途軸**（WiFi 等）。両者の照合には「期待対応表」
   （用途ラベル → 期待される方式クラスの集合）を使うが、これは **[仮説]** であり、
   一致率を「精度 (accuracy)」と呼ぶことを禁止する（レポートのバナーに明記）。
4. **実データは読み取り専用**: captures/ 内のファイルの移動・改名・上書き禁止。
   隔離済みの captures/_review_pending/ は対象外（含めない）。
5. **不一致は故障ではなく観測**: 期待と違う予測は、
   (a) ドメインギャップ（Sim と実の見え方の差）
   (b) 13ms くじ引き（バーストが窓に写っていない）
   (c) ルールラベル自体の誤り
   の3因に分解して読む。レポートと完了報告はこの3因の観点で見立てを書く。

## 事前確認（実装前に必ず行い、結果を報告に含める）
1. `.gitignore` に `simdata/` と `runs/` が入っているか確認。無ければ追加して報告
   （プローブ出力や学習成果物がリポジトリを汚さないため）。
2. `dataset.load_index("captures/")` が
   - サブディレクトリ `_review_pending/` を**含まない**こと
   - 件数（期待: 49）と hw が "HackRF One" のみであること
   を確認。挙動が期待と違えば（例: サブディレクトリも拾う）、対象の絞り方を
   決めて報告（captures/ 直下のみに限定する等。データ自体は動かさない）。
3. 実データに存在する**ラベルの一覧と件数**を列挙（`python -m dataset stats captures/`
   相当）。期待対応表がこの全ラベルをカバーするようにする。
4. チェックポイントの用意: M1 の成果物はクリーンアップ済みのため、固定シードで
   `simgen`（per-class 60）→ `train`（8 epoch）を再実行し `runs/m1/` を作る（約1分）。
   sim val 精度が M1 報告値（91.7%）と概ね整合（±数%）することを確認して報告。

## 実装内容

### (1) 期待対応表（[仮説]マーク必須）
`cnntrain/probe.py`（または classes.py）に、用途ラベル → 期待方式クラス**集合**の
対応表を定義する。例（あくまで叩き台。事前確認3で見つかった全ラベルを
カバーするよう、根拠つきで最終化すること）:
- "WiFi (2.4GHz, 20/40MHz)" → {wideband-ofdm}
- "BLE/Bluetooth (adv?)" → {narrowband-burst}（バースト未着なら noise-only が出うる
  ＝くじ引きの証拠としてカウント）
- "Zigbee/独自2.4G" → {narrowband-burst, wideband-ofdm}（帯域幅がまたがるため集合で）
- "WiFi 5G W56 (DFS)" → {wideband-ofdm}（実画像は細線が多く cw-tone が出る可能性に
  言及してよい＝ラベル監査の観点）
- "5G NR n79 4.7G" → {wideband-ofdm}
- "航空監視レーダ等" → {pulse-radar}
対応表には「これは地図上の仮説であり、画像確認と将来の人間レビューで更新する」
旨のコメントを必ず付ける。対応表にないラベルは "unmapped" として集計する。

### (2) プローブ CLI
`python -m cnntrain.probe --data captures/ --checkpoint runs/m1/checkpoint.pt --out runs/probe_real/`
- 各レコード: 読み込み → spec.render → 推論 →（ファイル名, 実ラベル, 予測クラス,
  確信度）を記録。
- 集計:
  - 実ラベル × 予測クラスの**クロス表**（件数）
  - ラベル別の期待一致率（対応表ベース）と平均確信度
  - ラベル別の **noise-only 率**（くじ引き問題の定量値。特に BLE）
- **要画像確認リスト**: 期待と不一致のうち、確信度の高い順に上位 N 件（既定 10）の
  ファイル名を列挙（ユーザーが captures/_images/ の PNG と突き合わせるため）。
- レポートを text + JSON で `--out` に出力。冒頭バナー（必須・eval-harness 流儀）:
  - SIM-TRAINED MODEL / REAL-DATA PROBE（合成データのみで学習したモデルによる
    実データ探針である）
  - 期待対応表は [仮説] である
  - **一致率は accuracy ではない**（軸が違う照合であり、3因分解で読むこと）

### (3) テスト（高速・実 captures 不要）
- simgen の合成データを実データの代役にして、プローブの**機構**
  （クロス表集計・期待照合・unmapped 扱い・レポート生成）を検証する 1〜2 本。
- 既存テストの変更・削除は不可（追加のみ）。新規テストは数十秒以内。

## 検証（実装後に必ず行う）
1. `python -m py_compile` 対象ファイル → OK。
2. `python -m pytest -q` → 既存 109 passed 維持 + 新規分 全緑。
3. 凍結契約 diff 空: `git diff --stat -- spec.py sigmf_io.py`。
4. 実データでプローブ実走: captures/ 49 件 → レポート生成まで完走。所要時間を報告。
5. captures/ のファイルが1つも変更・移動されていないこと（git status と
   ファイル数で確認。captures/ は gitignore 済みなので件数確認を明示的に）。

## 完了報告に含めること
- 事前確認の結果（gitignore / load_index の挙動と件数 / ラベル一覧 /
  チェックポイント再現の精度）。
- 最終化した期待対応表（各行の根拠つき）。
- **クロス表**（実ラベル × 予測クラス）と、ラベル別の期待一致率・noise-only 率・
  平均確信度。
- 要画像確認リスト（上位件のファイル名と、なぜ面白いかの一言）。
- **3因分解の見立て**: 結果のうち、どれが (a) ドメインギャップ /
  (b) 13ms くじ引き / (c) ラベル誤り候補 に見えるか。根拠とともに。
- バナーの所在。テスト結果（N passed）・凍結 diff 空・captures/ 無変更の明言。
- 将来課題（実データ fine-tune の是非、バースト同期キャプチャの優先度への示唆、
  classify 接続への含意 等）。

## やってはいけないこと（禁止事項の再掲）
- 実データでの学習・fine-tune・しきい値調整（推論のみ）。
- captures/ 内のファイルの移動・改名・上書き・削除。_review_pending/ を対象に含める。
- spec.py / sigmf_io.py の変更。spec.render を迂回する前処理。
- classify.classify への接続・変更。
- 既存テスト・既存 cnntrain モジュールの破壊的変更（probe 追加と対応表追加のみ。
  既存の simgen/train/evaluate/infer の挙動は変えない）。
- 一致率を「精度」「accuracy」と表現すること（バナー・レポート・報告のすべてで）。
