# 作業指示書: capture-engine — 帯域幅(BW)測定の修正（案1＋案3・対症療法）

## 役割
あなたは sigscan プロジェクトの capture-engine エージェント。
本指示は、先の調査で機序が確定した「BW 測定バグ」に対する**対症療法の修正**。
2つだけ直す:
- **案1**: `measure_signal` の `bw_hz` を「活性ビンの最外殻の差(max-min span)」から
  「主ピークを含む連続占有帯の幅」へ（2信号を跨いで膨張するのを止める）。
- **案3**: `bw_median_hz` の母集団を「全観測」から「信号が検出された観測」に限定
  （合間の CW-only 窓に潰れるのを止める）。

**これは対症療法であり根治ではない。** 根因は「1窓に複数信号が同居している」ことで、
その根治（複数信号を各々分離して測る）は返り値の形を変える設計判断＝task#6（マルチ信号）で扱う。
本タスクは**返り値の形（凍結シグネチャ）を保ったまま、値の出方だけを正す**。

## 背景（調査で確定済みの事実）
- `measure_signal`（dsp.py:129 付近）は `bw = active.max() - active.min()`。占有帯域でなく
  「床+6dB を超えた活性ビンの最外殻の差」なので、窓内に離れた2信号があると**間の不活性ノイズごと跨ぐ**
  （再現: 占有 8.2% なのに 10.78MHz）。この値が edge/rep 系に流れ、classify の用途判定を誤らせる
  （14.55MHz → ISM2.4G ルールで「Zigbee 仮説」→ (C) C-conflict の一因）。
- `bw_median_hz`（dwell.py:133 付近）は全観測の `measure_signal.bw_hz` の median。BLE は間欠で、
  滞在中の**過半の観測は「2425MHz の細い CW 線だけ」**が活性 → その窓で幅が ~1 FFT ビン
  （20e6/1024 = 19531.25Hz）に潰れ、median が多数派の quiet 窓に支配される。
- `bw_median_hz` は品質ゲート（quality.py の極細スプリアス/comb 判定）専用で、classify には流れない。
- **classify の帯域幅しきい値（BLE<3MHz / Zigbee 3–15MHz / WiFi 15–45MHz 等）は本タスクでは変更しない**
  （区切りの見直しは、正しく測り直したデータが揃ってから別途・人間承認で行う。理由: 一度に一つだけ直す）。

## 最重要原則（絶対厳守）
1. **凍結契約不可侵**: `spec.py` / `sigmf_io.py` は変更しない。`dsp.measure_signal` /
   `dsp.detect_segments` は6継ぎ目 → **引数・返り値の形（キー構成・型）を変えない**。
   `bw_hz` は `bw_hz` のまま（値の計算だけ変える）。`bw_median_hz` も同様。
   `git diff --stat -- spec.py sigmf_io.py` が空であること。
2. **classify のしきい値は変更しない**: classify.py の帯域幅の区切り数値（3MHz/15MHz/45MHz 等）に
   一切触れない。今回直すのは「測る側(dsp/dwell)」だけ。
3. **detect_segments の橋渡し併合（案2）は今回やらない**: survey 側の併合ガードは別タスク（保留）。
   本タスクは案1（measure_signal）と案3（bw_median の母集団）の2点のみ。
4. **最小実装・スコープ厳守**: 上記2点以外は直さない。マルチ信号分離・複数信号の返却など
   返り値の形を変える改造は禁止（task#6）。迷ったら最小実装。
5. **読み取り専用データ**: captures/ の `*.sigmf-*` を書き換えない。診断用の一時ファイルは
   captures/ 外・非コミット（scratchpad 等）。メタを読むなら sigmf_io の cp932 経路（UTF-8 決め打ち禁止）。

## 実装内容

### 案1: measure_signal の bw_hz を「連続占有帯」へ（dsp.py）
- `bw = active.max() - active.min()` を、**主ピーク（最大パワーのビン）を含む連続活性ラン**の幅に変更。
  - 具体的方針（実装者判断で最小に）: 床+6dB マスクの上で、**ピークビンから両隣へ連続してマスクが
    True である範囲**を辿り、その連続区間の幅を `bw_hz` とする（間に不活性ビンが挟まったら止める）。
  - これにより「2425 CW 線〜BLE バースト」を跨ぐ 10.78MHz が、主ピーク側の信号の実幅（~1.5MHz 等）に収まる。
- **既存の広帯域分岐は保持**: `occupied_frac > 0.85 → bw = rate` はそのまま
  （真に窓を埋める WiFi 等を過小評価しないため）。案1 は「離れた2山を跨ぐ」ケースだけを是正し、
  連続する広帯域は従来どおり広く測れること。
- 返り値のキー・型・その他の項目（center, snr 等）は不変。`bw_hz` の**値だけ**変える。
- trade-off の明記（調査どおり）: 中程度に離れた2山があるとき、主ピーク側**1つ**の幅を返す
  （もう一方は取りこぼす＝1値で返す構造上の限界。根治は task#6）。この妥協を完了報告に記す。

### 案3: bw_median_hz を「検出された観測」に限定（dwell.py）
- median の母集団を、全観測の bw から **検出された観測（既存の検出しきい/マージンを満たす観測）**の
  bw に限定する。既存の `det_snr`（dwell.py:111 付近）等、**既にある指標を再利用**（新たな検出器を作らない）。
- **フォールバック必須**: 検出された観測が 0 件のとき（対象が終始 quiet 等）は、
  従来どおり全観測の median にフォールバックする（None や例外にしない）。この分岐を明示。
- 返り値のキー `bw_median_hz` の形・型は不変。値の代表性だけ正す。

### テスト（下記ルールに従う）
本タスクは**既存テストの期待値が変わり得る例外的タスク**。次を厳守:
- **シグネチャ系（tests/test_seams.py）は絶対に変えない**。緑のまま維持（返り値の形を保つ証拠）。
- **案1/案3 の新しい挙動を固定する追加テストを新規で書く**:
  - 案1: 「離れた2信号（細い CW + 狭バースト）を含む合成 IQ で、bw_hz が主ピーク信号の実幅に収まり、
    2信号を跨がないこと」。および「連続する広帯域(occupied>0.85)で従来どおり bw≈rate になること」（非退行）。
  - 案3: 「間欠バースト＋常時 CW の多観測列で、bw_median が検出観測に基づく値になり、
    CW-only 窓に潰れないこと」。および「検出 0 件で全観測 median にフォールバックすること」。
- **既存テストの期待値変更は、必要な場合のみ・最小限**。変更する場合は「どのテストを・なぜ・
  旧期待→新期待」を完了報告に列挙（勝手に消さない・弱めない）。単に古い bw 定義に依存していた
  アサーションを新定義に合わせる範囲に留め、テストの意図は保つ。
- quality 系（極細スプリアス/comb）の期待が bw_median の代表性変化で動く場合も、同様に最小修正＋報告。
