# バンドプラン追加提案 — 1〜6GHz「完全な地図」化

**目的**: 現行 `config.BAND_PLAN`(通信偏重・17帯域)に、レーダ・気象・衛星・特殊業務など
の「穴」を追加し、1〜6GHz に実在する信号の **完全な地図** を作る。

**重要な前提（地図と踏破の分離）**:
- 本提案は **地図作り（全体設計）**。ここに載せる＝今すぐ収集する、ではない。
- 実際の **踏破（データ収集）** は 2.4GHz 帯など受信実績のある帯域から着手する。
- 受信できない帯域も「そこに何が存在するはずか」を記録する。これが Unknown 信号の
  正体を推定する際の **強力なコンテキスト** になる（例: 5.25-5.37GHz に気象レーダが
  あると地図にあれば、5.3GHz 付近のパルスを気象レーダと推定できる）。

**一次情報の出典**:
- 総務省「我が国の電波の使用状況」令和7年3月1日時点
  (https://www.tele.soumu.go.jp/resource/search/myuse/use/ika.pdf)
- 周波数割当計画（令和6年総務省告示第402号, 令和6年12月16日）
- ※各周波数・用途はこの公式資料の記載に基づく。決め打ちではなく一次情報が根拠。

## 1. ラベル体系（確定済み・再掲）

各信号には **用途(use)** と **方式(modulation)** の2つのラベルを、それぞれ確信度つきで付ける。
確信度 < 0.7 → 用途は `Unknown` 扱い（候補は内部保持し、後で review.py で人間が判断）。

### 用途ラベル（粗め・初期）
`WiFi` / `Bluetooth` / `Zigbee` / `Cellular` / `SatComm` / `GNSS` / `Radar` /
`ETC-DSRC` / `Drone-FPV` / `Amateur` / `WPT` / `Spurious` / `Unknown`

> 注: `Radar` は初期は1つに統合（気象/航空監視/船舶の細分化はしない）。
> 細かな用途は「周波数帯 × 方式ラベル」の掛け合わせで後から推定する。
> `WPT`（ワイヤレス電力伝送）は今回の一次情報調査で 2.4GHz 等に存在が判明したため追加。
> **`Spurious`（受信機内部スプリアス）は 2026-07-15 追加**。HackRF の 40MHz クロック高調波
> (2400/2440/2480MHz)・16MHz コム等は「電波」ではなく受信機が生む偽信号だが、実RFに常在し
> CNN が最も誤認するため、**明示的に1クラスとして学習させる**（`spurious` として ground truth
> を32件確定済み）。これは机上の一次情報には無く、実地の収集で発見したもの。専門家CNNは
> これを「実信号ではない」と弾くために本クラスを持つ。詳細は `docs/ARCHITECTURE.md §9`・
> `CLAUDE.md §7`（HackRF 固有スプリアス）。

> **2.4GHz 専門家の当面のクラス（案Z・2026-07-15 確定）**: 上記のうち用途1軸で
> `ble-adv`（=Bluetooth の下位）/ `wifi-24`（=WiFi 2.4G）/ `spurious` / `unknown` を採用。
> `unknown` は教師0件のため当面3クラスで開始し蓄積後に追加。方式軸・2軸化は将来課題。

### 方式ラベル（粗め・初期）
`OFDM` / `FHSS` / `DSSS` / `Chirp-LFM` / `FMCW` / `Pulse` / `PhaseCoded` /
`CW-Narrow` / `Unknown`

> 拡張の器（将来）: 規格(standard 例 802.11n)・変調次数(order 例 64QAM) などは
> `sigscan:` 名前空間に後から追加。今は空でよい。

## 5. 反映方法（あなたが承認後に行う作業）

1. **config.py の BAND_PLAN に追加**: 上記 A/B/C の Band を追記。
   - `f_lo`/`f_hi` は MHz×1e6 = Hz で記述（例: 2700e6）。
   - 既存「Aero/Radar S-band 2.7-2.9」は f_hi を 3400e6 に拡張（修正R3）。
2. **ラベル候補のマッピング**: 各 Band の hint に「想定用途/方式」を併記、または
   別途 `BAND_USE_HINT` のような対応表を持たせる（classify が用途推定に使う）。
3. **X注記（混入要因）の扱い**: quality.py / classify.py が
   「2.4GHz の強い定常信号は WPT 候補」「DFS帯の周期パルスは気象レーダ候補」と
   判定できるよう、注記をルールやコメントに反映（将来）。
4. **踏破は据え置き**: バンドプラン拡張後も、収集は 2.4GHz から（§4 の◎）。

## 6. 未確定・要検討事項（次の議論用メモ）

- **専門家(AI)の単位**: 地図は細かいが、専門家は周波数で束ねる。境界は 2.4GHz 専門家を
  完成させてから、実地の知見で確定（"検証してから決める"）。
- **HackRF 20MHz 窓との整合**: 広帯域(5G n77/n78 は100MHz級)を1専門家で扱うか分割か。
- **方式の自動判定**: Chirp-LFM / FMCW / Pulse の見分けは CNN 学習で要検証（斜め線・
  のこぎり・周期縦線の弁別）。
- **priority の最終調整**: 受信現実性(§4)を反映して再設定。

## 7. 1GHz以下への地図延長（指示書「1GHz以下の受信対応」Part B・実装済み 2026-07-23）

**経緯（1行）**: 本提案・`config.BAND_PLAN` は当初 **1〜6GHz で範囲を切っていた**ため、
FM放送・LPWA 等の **1GHz以下が丸ごと未記載**で、受信しても全て「未識別信号(バンドプラン外)」
表示になっていた。アンテナ `Diamond D1300AM` の受信範囲は 25-1300MHz あり実信号が届くため、
地図を 1GHz以下へ延長した。

**追加した帯域**（`config.BAND_PLAN` に entries 追加のみ。`Band` dataclass 定義は不変。
全て `f_hi<=928MHz` で既存最下限 960MHz より下＝**既存31バンドと周波数が重ならない**）:

| バンド | 範囲(MHz) | 用途/方式(地図・仮説) | 備考 |
|---|---|---|---|
| FM放送 | 76-95 | Broadcast / FM | 国内FM(V-Low含む)。連続・広帯域・常時 |
| 航空無線(VHF/AM) | 118-137 | Aero-Voice / AM | 航空管制VHF音声。D1300AM は AM 対応 |
| 業務/防災無線(VHF) | 150-174 | LandMobile / FM,Digital | 陸上移動業務・消防防災等 |
| 特定小電力 315MHz帯 | 312-316 | SRD / ASK,OOK | 300MHz帯特定小電力+315MHz RKE。※提案315-316を312-316へ補正(実割当312-315.25に整合) |
| 特定小電力 426MHz帯 | 426-429.75 | SRD / FM,Digital | 426/429MHz帯。※提案426-430をアマチュア430との重複回避で426-429.75へ補正 |
| アマチュア 430MHz帯 | 430-440 | Amateur / FM,SSB,Digital | 国内アマチュア70cm帯 |
| 地上デジタルTV(UHF) | 470-710 | Broadcast / OFDM(ISDB-T) | 地デジ物理ch13-52。連続・広帯域・常時 |
| LPWA 920MHz帯 | 920-928 | LPWA,RFID / LoRa,FSK,Wi-SUN | 国内920MHz帯(RFID/センサ網)。間欠・狭帯域 |

- **priority は全て 1**（控えめ）: 既定 1-6GHz 巡回への影響を最小化。1GHz以下の運用は
  `--focus` + `--start/--stop` で当該帯域に張り付く想定。
- **ラベル**: 新バンド名は `review.py._available_labels()` が BAND_PLAN から自動収集する
  ため、人手○×UIの選択肢に自動で並ぶ（明示的なラベル追加は不要）。rule 分類は用途DB
  (`SIGNAL_DB`)に該当規則が無いため確信度 0.30-0.45（<0.7）→ Unknown 経由で **人間確定**へ
  （量より質。用途/方式の精緻化ルールは将来課題）。
- **運用注記**: 1GHz以下では LNA(2.4GHz専用フィルタ内蔵)を**外す**（コード変更不要の運用ルール）。
- **併せて実装（Part A）**: ゼロIF受信機の DC 残留がチューナ中心に張り付き本物の放送局を
  奪う問題への opt-in ガード `--dc-guard-hz`（既定0=無効）。dwell 測定でチューナ中心±HZ を
  主役候補から外し次に強い本物を拾う。詳細は `指示書_1GHz以下の受信対応.md`。
