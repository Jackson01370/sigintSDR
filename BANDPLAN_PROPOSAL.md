# バンドプラン追加提案 — 1〜6GHz「完全な地図」化

**目的**: 現行 `config.BAND_PLAN`(通信偏重・17帯域)に、レーダ・気象・衛星・特殊業務など
の「穴」を追加し、1〜6GHz に実在する信号の **完全な地図** を作る。

**重要な前提（地図と踏破の分離）**:
- 本提案は **地図作り（全体設計）**。ここに載せる＝今すぐ収集する、ではない。
- 実際の **踏破（データ収集）** は 2.4GHz 帯など受信実績のある帯域から着手する。
- 受信できない帯域も「そこに何が存在するはずか」を記録する。これが Unknown 信号の
  正体を推定する際の **強力なコンテキスト** になる（例: 5.25-5.37GHz に気象レーダが
  あると地図にあれば、5.3GHz 付近のパルスを気象レーダと推定できる）。

**一次情報の出典**:
- 総務省「我が国の電波の使用状況」令和7年3月1日時点
  (https://www.tele.soumu.go.jp/resource/search/myuse/use/ika.pdf)
- 周波数割当計画（令和6年総務省告示第402号, 令和6年12月16日）
- ※各周波数・用途はこの公式資料の記載に基づく。決め打ちではなく一次情報が根拠。

---

## 1. ラベル体系（確定済み・再掲）

各信号には **用途(use)** と **方式(modulation)** の2つのラベルを、それぞれ確信度つきで付ける。
確信度 < 0.7 → 用途は `Unknown` 扱い（候補は内部保持し、後で review.py で人間が判断）。

### 用途ラベル（粗め・初期）
`WiFi` / `Bluetooth` / `Zigbee` / `Cellular` / `SatComm` / `GNSS` / `Radar` /
`ETC-DSRC` / `Drone-FPV` / `Amateur` / `WPT` / `Unknown`

> 注: `Radar` は初期は1つに統合（気象/航空監視/船舶の細分化はしない）。
> 細かな用途は「周波数帯 × 方式ラベル」の掛け合わせで後から推定する。
> `WPT`（ワイヤレス電力伝送）は今回の一次情報調査で 2.4GHz 等に存在が判明したため追加。

### 方式ラベル（粗め・初期）
`OFDM` / `FHSS` / `DSSS` / `Chirp-LFM` / `FMCW` / `Pulse` / `PhaseCoded` /
`CW-Narrow` / `Unknown`

> 拡張の器（将来）: 規格(standard 例 802.11n)・変調次数(order 例 64QAM) などは
> `sigscan:` 名前空間に後から追加。今は空でよい。

## 5. 反映方法（あなたが承認後に行う作業）

1. **config.py の BAND_PLAN に追加**: 上記 A/B/C の Band を追記。
   - `f_lo`/`f_hi` は MHz×1e6 = Hz で記述（例: 2700e6）。
   - 既存「Aero/Radar S-band 2.7-2.9」は f_hi を 3400e6 に拡張（修正R3）。
2. **ラベル候補のマッピング**: 各 Band の hint に「想定用途/方式」を併記、または
   別途 `BAND_USE_HINT` のような対応表を持たせる（classify が用途推定に使う）。
3. **X注記（混入要因）の扱い**: quality.py / classify.py が
   「2.4GHz の強い定常信号は WPT 候補」「DFS帯の周期パルスは気象レーダ候補」と
   判定できるよう、注記をルールやコメントに反映（将来）。
4. **踏破は据え置き**: バンドプラン拡張後も、収集は 2.4GHz から（§4 の◎）。

## 6. 未確定・要検討事項（次の議論用メモ）

- **専門家(AI)の単位**: 地図は細かいが、専門家は周波数で束ねる。境界は 2.4GHz 専門家を
  完成させてから、実地の知見で確定（"検証してから決める"）。
- **HackRF 20MHz 窓との整合**: 広帯域(5G n77/n78 は100MHz級)を1専門家で扱うか分割か。
- **方式の自動判定**: Chirp-LFM / FMCW / Pulse の見分けは CNN 学習で要検証（斜め線・
  のこぎり・周期縦線の弁別）。
- **priority の最終調整**: 受信現実性(§4)を反映して再設定。
