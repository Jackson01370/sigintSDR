# 作業指示書: cnn-training M2.5 — simgen 形状の実測再アライン（第2周）

## 役割
あなたは sigscan プロジェクトの cnn-training エージェント。
本指示は、M2 再プローブの 9 枚のスポットライト画像を**人間が確認**して得た
「形状ギャップの定量値」を simgen に反映し、再学習 → 再プローブで
「在窓 BLE → narrowband-burst」「実 WiFi → wideband-ofdm」への収束を**数字で測る**
第2周の実測アラインである。パイプラインは既存のまま、**生成パラメータの修正が主**。

## 背景（画像確認で測れた形状ギャップ — これが設計の根拠）
M2 のスポットライト9枚（WiFi 8 / BLE 1）を人間が確認した結果:
1. **実 WiFi の形**（8枚で一致）:
   - 帯域: **キャプチャ全幅（20MHz, 画面の上端〜下端）を埋める**。
     → 現行 wideband-ofdm の 12〜16MHz（画面の6〜8割・上下に余白）は**狭すぎ**。
       実 WiFi はむしろ pulse-radar（≈18MHz 全面）の帯域に近い。
   - 1 本の縦縞の幅: 約 **0.4〜0.8ms**（13ms 窓 ≈ 262144 サンプルに対し 3〜6%）。
   - 時間配置の2パターン: 「**2 本だけ + 長い沈黙**」または
     「**密集クラスタ + 死んだ時間（無信号区間）**」。現行の窓全体への
     均等散布は実態と違う。
2. **実 BLE の形**（Image 1）:
   - 帯域幅 ≈ **2MHz**（M2 の 1〜2.5MHz は妥当・**維持**）。
   - 1 発の長さ: 約 **0.4〜0.5ms**（窓の ~3%）と**短い**。窓に **1〜3 発**だけ。
     → 現行 narrowband-burst より短く・疎ら。これが「在窓 BLE が
       narrowband-burst に届かず cw-tone 止まり」の一因（形が合っていない）。
3. **中心外の細い持続線**（Image 6/7/8/9 に実在）:
   - 画面中央**ではない**位置（例 2413.5/2432.1MHz）の細い横線 = **実在の
     スプリアス or 狭帯域 CW**（DC ではない）。M2 で再定義した「cw-tone=中心外の
     細線」に合致するため、**残存する cw-tone 誤予測の一因**の可能性。
   - 対策: DC と同じ思想で、**中心外スプリアス線も全クラスに無相関で注入**し、
     「単独の細線が在るだけでは cw-tone と断定しない」を学べるようにする。
4. **マルチ信号の同居**（Image 4: WiFi 縦縞 + BLE ブロブ）:
   - 実画像は複数信号が同一窓に共存。合成は 1 クラス 1 枚。
   - **本 M2.5 では対応しない**（構造的単純化は M3 以降の課題として記録のみ）。

## 最重要原則（絶対厳守）
1. **凍結契約不可侵**: spec.py / sigmf_io.py 変更禁止。スプリアス線・バーストの
   注入は **IQ（時間領域）合成**で行い凍結 spec.render を通す（画像描き込み禁止）。
2. **実データで学習・fine-tune しない**（推論のみ）。ラベル不使用の物理量計測
   （帯域占有・スプリアス線強度などを実レコードから測り較正に使う）は許可。
3. **captures/ の *.sigmf-* は読み取り専用**（移動・改名・上書き禁止）。
   スポットライト出力 captures/_images/_spotlight/ への書き込みは可。
4. **公平な before/after**: M1/M2 と同一条件（per-class 60 / seed 0 / 8 epoch /
   同モデル）で再学習し、**M2 ベースライン（下記転記）と比較**する。
5. **pulse-radar の定義は変えない**: 「厳密周期・短パルス・一様幅」を維持する
   （wideband-ofdm を全帯域化すると pulse-radar と帯域が同じになるため、
   **両者の弁別は"時間構造"——周期 vs 非周期/クラスタ——に委ねる**。この対比が
   崩れていないか生成統計で必ず確認）。
6. **チューニング無限ループ禁止**: 実装 → 学習 → プローブ → 報告で1サイクル。
   明白なバグ（sim val 崩壊・注入位置誤り等）のみ修正して**1回だけ**再実行可。
   それ以上の精度狩りはしない（次サイクルの判断は人間）。

## 実装内容

### (1) narrowband-burst の形状修正（在窓 BLE 実測へ）
- 帯域幅は維持（1〜2.5MHz）。**バースト長を短く（≈0.3〜0.6ms 相当）**、
  **窓内の本数を 1〜3 発**に。現行が長め/多めなら実測へ寄せる。
- 目的: 在窓 BLE が cw-tone ではなく narrowband-burst に分類されるようにする。

### (2) wideband-ofdm の形状修正（実 WiFi へ）
- **帯域を全幅化**: 12〜16MHz → **キャプチャほぼ全幅（≈18〜20MHz 相当）**。
- **縦縞 1 本の幅 ≈ 0.4〜0.8ms**（窓の 3〜6%）。
- **時間配置を2モードに**: ランダムに
  (i) 「少数（2〜4 本）+ 長い沈黙」 か (ii)「密集クラスタ + 無信号区間」 を選ぶ
  （実観測の2パターンを反映）。窓全体への均等散布はやめる。
- **pulse-radar との弁別維持**: pulse-radar は厳密周期。wideband-ofdm は
  非周期（本数・間隔・クラスタの有無がランダム）。生成統計（間隔の変動係数 CV、
  クラスタ性）で両者が分離していることをテスト確認。

### (3) 中心外スプリアス線の注入（全クラス無相関）
- DC 注入（中心固定）に加え、**中心から離れた位置（ランダムなオフセット）に
  細い持続 CW 線**を、確率 p（小さめ、例 0.2〜0.4）で**全クラスに無相関**に注入。
- 強度・本数は事前確認4の実測に較正。**クラスと無相関**（cw-tone クラス専用に
  しない）。これで「中心外の単独細線が在る ≠ 直ちに cw-tone」を学べる。
- 注意: cw-tone クラスの"主信号"は引き続き中心外の細線だが、他クラスにも
  薄いスプリアス線が混じることで、CNN は「主たる構造」で判断するようになる
  （狙い: BLE/WiFi に混じる細線に引きずられないこと）。
- メタに sigscan:spur_injected 等を記録（検証可能化）。

### (4) 再学習・sim 評価・実プローブ・スポットライト
- 新 simdata 生成（per-class 60, seed 0）→ runs/m2_5/ に学習（8 epoch, seed 0）。
- sim val の精度・混同行列を報告（下がっても正直に。崩壊＝単一クラス潰れのみ要修正）。
- captures/（64件）で再プローブ → runs/probe_real_m2_5/。
- スポットライトも再実行（captures/_images/_spotlight/ を上書き）。

### (5) before/after 比較（M2 → M2.5）
M2 ベースライン（転記）:
- BLE(49): cw-tone 24 / noise-only 25 / narrowband-burst 0。期待一致 0%。noise-only率 51%。
- WiFi(13): pulse-radar 9 / wideband-ofdm 1 / cw-tone 2 / noise-only 1。期待一致 7.7%。
- Zigbee(2): cw-tone 1 / noise-only 1。
観測すべき方向（仮説・外れても正直に報告）:
- 在窓 BLE（noise-only でない約半数）が cw-tone → **narrowband-burst** に動くか。
- WiFi が pulse-radar → **wideband-ofdm** に動くか（時間構造の違いで分離できるか）。
- noise-only 率（くじ引き）は形状修正では基本下がらない見込み（下がるなら理由を考察）。

### (6) テスト（既存無変更・追加のみ・高速）
- T9: narrowband-burst のバースト長・本数が実測域（短く 1〜3 発）に入る。
- T10: wideband-ofdm が全幅・2モード（少数+沈黙 / クラスタ）で、pulse-radar の
  周期性と統計的に分離している（CV 等）。
- T11: 中心外スプリアス注入が全クラスに無相関で入り、強度が較正域内。
- 既存テスト（特に M2 の T5–T8、DC 注入・cw オフセット）が**意図変更で落ちる場合は
  勝手に書き換えず中断・報告**。pulse-radar 関連テストを壊さないこと。
