# 作業指示書: capture-engine — view_captures のラベル表示修正（+ 帯域幅測定の怪しさを記録）

## 役割
あなたは sigscan プロジェクトの capture-engine エージェント。
本指示は主に **小さな整備作業**: `view_captures.py` が画像一覧・PNG タイトルで
ラベルを `(no label)` としか出さない穴を塞ぎ、SigMF メタの用途ラベルと
（あれば）CNN 監査の verdict を表示する。あわせて、別件の既知バグ
「ルールの帯域幅測定が画像の実体と食い違う」を**コード変更せず記録だけ**する。

## 背景
- 現在 captures/ は 78 PNG まで増えたが、一覧・タイトルが全て `(no label)` で、
  どの画像が何のラベルかを目視で突き合わせられない（画像確認の規律の効率を下げている）。
- 直近の実機 run（M3・CNN監査つき）で、SigMF メタには各記録の
  用途ラベル（annotation）と CNN 来歴（global の `sigscan:cnn_verdict` 等）が
  載るようになった。これらをタイトルに出せば、一覧が「歩ける地図」になる。
- 別件（本指示では直さない）: ある 2433.49MHz / BW14.6MHz とラベルされた記録の
  実画像は「中心からずれた ~1.5MHz の小ブロック + 2425MHz 付近の細い持続線」で、
  **14.6MHz という帯域幅測定が画像の実体と大きく食い違う**ことが画像確認で判明した。
  ルールの bw 過大測定の疑い（過去にも bw_median 値への違和感が申し送られている）。

## 最重要原則（絶対厳守）
1. **凍結契約不可侵**: spec.py / sigmf_io.py は変更しない。`view_captures.render_one`
   が cnntrain.spotlight 等から再利用されているため、**既存のシグネチャと
   描画結果（スペクトログラム本体）を壊さない**。変えるのは「タイトル/一覧の
   文字列」だけ。
2. **読み取り専用**: captures/ の *.sigmf-data / *.sigmf-meta を一切書き換えない
   （ラベルはメタから読むだけ）。PNG 出力先（captures/_images/）への書き込みのみ可。
3. **メタ読み込みは既存の作法に従う**: SigMF メタは locale/cp932 で開く既存の
   read 経路（sigmf_io）を使う。UTF-8 決め打ちで開かない。
4. **後方互換**: CNN 来歴が無い古い記録（`sigscan:cnn_verdict` 不在）でも
   エラーにならず、ラベルだけ出す（verdict 欄は省略 or 空）。ラベルすら無い
   記録は従来どおり穏当に扱う（`(no label)` のままで可）。
5. **最小実装**: 表示の改善に限定。スコープを広げない（フィルタ機能・並べ替え等の
   新機能は作らない。やりたくなったら将来課題コメントに留める）。
6. **帯域幅測定バグは直さない**: 本指示では dsp / classify に触れない。
   下記「記録タスク」で**文書化のみ**。

## 実装内容
### (1) view_captures のラベル表示
- 一覧行と PNG タイトルに、メタから読んだ**用途ラベル**を出す
  （例: `2433MHz_..._1.png  2418.7MHz  Zigbee/独自2.4G`）。
- CNN 来歴があれば **verdict と方式クラスも併記**
  （例: タイトル末尾に `[CNN:C-conflict cnn=narrowband-burst@0.51]` のように）。
  無ければ省略（後方互換）。
- ラベルが取れない記録は従来どおり `(no label)`（穏当なフォールバック）。
- PNG 画像内のタイトル文字列にも反映（既存の matplotlib タイトル生成箇所に
  ラベル文字列を足す。スペクトログラム本体・軸・mean spectrum パネルは不変）。
- 日本語ラベル（"Zigbee/独自2.4G" 等）が PNG・コンソールで化けないこと
  （コンソールは既存の UTF-8 強制、PNG は matplotlib のフォント既定で可。
  豆腐になる場合のみ最小の対処を報告）。

### (2) 帯域幅測定の怪しさを「記録」（コード変更なし）
- リポジトリ内の適切な場所（例: `FILES.md` の既知課題欄、または
  `docs/` か `NOTES`/`TODO` 系の既存文書。無ければ最小の `KNOWN_ISSUES.md` 新規）に、
  以下を簡潔に記録する:
  - 症状: 2433.49MHz / ラベル BW14.6MHz の記録の実画像は ~1.5MHz の小ブロック +
    2425MHz の細い持続線で、**測定 BW が実体と大きく食い違う**。
  - 疑い: `dsp.measure_signal` / `dsp.detect_segments` がノイズ床や中心外スプリアス線を
    巻き込んで帯域幅を過大評価している可能性。過去の bw_median 違和感とも整合。
  - 影響: 過大 bw はルールの用途判定（帯域幅依存）を誤らせ、CNN 監査で (C) 化される
    一因になりうる（誤検出ではなく監査が正しく保留した例だが、根因は測定）。
  - 位置づけ: **未着手の調査課題**（本コミットでは記録のみ。修正は別タスク）。
- どこに記録したかを報告に明記。

### (3) テスト（既存無変更・追加のみ・高速）
- T: メタからラベル文字列を組み立てるヘルパ（純関数化できるなら純関数で）に対し、
  (i) 用途ラベル+CNN verdict あり、(ii) ラベルのみ（CNN来歴なし・古い記録）、
  (iii) ラベル無し（`(no label)` フォールバック）の 3 ケースを検証。
- `render_one` の既存テスト・呼び出し元（spotlight）が**無改変で緑**であること。
  描画結果の本体が変わっていないこと（タイトル文字列のみ変化）を確認。
