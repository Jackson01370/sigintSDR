# 作業指示書: capture-engine — 帯域幅(BW)測定バグの調査（機序解明・コード変更なし）

## 役割
あなたは sigscan プロジェクトの capture-engine エージェント。
本指示は **調査のみ（read-only・コード変更ゼロ）**。目的は、KNOWN_ISSUES.md に記録済みの
「ルールの帯域幅測定が画像の実体と食い違う」バグについて、**なぜそうなるのかを
コードから完全に解明し、根拠つきで報告する**こと。**修正はしない**（返り値の仕様が
未確定＝設計判断であり、かつ dsp は凍結継ぎ目のため。修正は人間承認後の別タスク）。

## 背景（実機実験で判明した事実 — これが最大の手がかり）
対象記録 `captures/2433MHz_1781101445472_7`（キャプチャ中心 2433.49MHz）について、
**画像確認＋実機でのルーター/イヤホン ON/OFF 実験**で以下が確定している:

- 画像には **2つの別々の信号**が写っている:
  1. **~1.5MHz 幅の狭いバースト**（実験で **Bluetooth（イヤホン）の BLE** と確定。周波数ホッピングで
     この付近を通過した瞬間を捉えたもの）。時間軸の一部にだけ出る短いバースト。
  2. **~2425MHz 付近の細い持続線**（全時間にわたる連続キャリア。BLE でも WiFi でもない別の常時オン信号。
     正体は未確定だが「バーストとは別物」であることは確実）。
- ところが SigMF メタの測定値は:
  - `annotation の edge 幅 (hi - lo) ≈ 14.55MHz`（**過大**）
  - `sigscan:bw_median_hz ≈ 19531.2 Hz`（**過小**。注: 20e6 / 1024 = 19531.25 とほぼ一致 = **1 FFT ビン幅**）
- **人間側の仮説（検証してほしい対象）**:
  - **edge 幅**は「BLE バースト（下側）〜2425 連続線（上側）＋間のノイズ床」を**1つの帯域として跨いで**
    掴んでいる（＝2信号を1信号と誤認）→ だから ~14.5MHz に膨らむ。
  - **bw_median** は逆に、何らかの理由で**中央値が 1 FFT ビンに潰れて**いる（ノイズ床の中央値を拾っている等）
    → だから ~19.5kHz に縮む。
  - **真の信号（~1.5MHz の BLE バースト）は、どちらの測定でも正しく捉えられていない。**

この仮説が正しいか、コードを読んで確定させるのが本タスク。

## 最重要原則（絶対厳守）
1. **調査のみ・コード変更ゼロ**: dsp.py / classify.py / spec.py / sigmf_io.py など一切編集しない。
2. **凍結継ぎ目の認識**: `dsp.measure_signal` / `dsp.detect_segments` は6継ぎ目で
   **シグネチャ（引数・返り値の形）変更禁止**。今回は調査のみだが、修正案を書く際も
   「返り値の形を保ったまま中身をどう直すか」という枠内で提案すること（仕様を壊す案は出さない）。
3. **読み取り専用**: captures/ の `*.sigmf-*` を書き換えない。診断用に一時ファイルを作る場合も
   captures/ の外（例: リポジトリ直下の使い捨て or /tmp）に置き、コミットしない。
4. **メタ読み込みは既存作法**: sigmf_io の read 経路（locale=cp932）で開く。UTF-8 決め打ち禁止。
5. **推測で断定しない**: 「たぶん」で片付けず、**実際のコード行と、実データでの数値**で裏を取る。
   仮説が外れていたら、外れていたと正直に報告する（真の機序を優先）。

## 調査内容（実施して報告に含める）

### (A) コード経路の特定
1. `dsp.py` を読み、次の2関数の**中身**を把握して報告:
   - `measure_signal(...)`: 何を入力に、何を返すか。`bw_median_hz` に相当する値を**どう計算しているか**
     （どの配列の median を、どの単位で取っているか。FFT サイズ・ビン幅の扱い）。
   - `detect_segments(...)`: 信号の帯域（lo/hi＝edge）を**どう決めているか**
     （しきい値の取り方、ノイズ床推定、連結の仕方。連続する周波数ビンをどう1区間にまとめるか）。
2. これらが使う FFT サイズ / ビン幅を報告（`spec.render` は nfft=512 だが、dsp が**別の nfft**を
   使っているなら、その値と 19531.2Hz = rate/nfft の対応を確認 → 例: 20e6/1024=19531.25 なら nfft=1024）。
3. classify 側で、この edge 幅 / bw_median がどう使われ、**どちらが用途判定（帯域幅依存のルール）に
   効いているか**を報告（今回 2433MHz が Zigbee 仮説→(C) になった経路との関係）。

### (B) 実データでの数値的裏取り
4. 対象記録の生 IQ（`captures/2433MHz_1781101445472_7.sigmf-data`）を読み、`measure_signal` /
   `detect_segments` を**そのまま呼んで**、以下を実測して報告（**読み取りのみ・書き戻さない**）:
   - detect_segments が返す区間（lo/hi、または複数区間があるか）。**区間が1つに繋がっているか、
     本来2つ（BLE バースト＋2425線）に分かれるべきところを1区間にしているか**を数値で示す。
   - measure_signal が返す bw_median 相当値と、その計算の途中経過（median を取っている配列の中身の
     ざっくりした分布。ノイズ床なのか信号なのか）。
   - 可能なら、周波数軸上で「どこからどこまでを1信号as見なしているか」を数値（MHz）で示す
     （例: lo≈2411MHz, hi≈2425.5MHz のように）。仮説「2信号跨ぎ」の成否を確定させる。
   ※ このデータが 0 バイト等で読めない場合は、読める別の実記録
     （例: A 記録 `captures/2433MHz_1781101180999_1`、または captures/ 内で BLE ラベルの生きた記録）で
     同等の検証を行い、どの記録を使ったか明記する。**データ欠損時は無理に捏造せず、読めた記録で代替。**

### (C) 機序の結論
5. 上記から、**edge が過大になる理由**と **bw_median が過小になる理由**を、それぞれ
   **コードの該当行を引用して**結論づける。人間の仮説（2信号跨ぎ／1ビン潰れ）が当たっているか、
   違うなら真の原因は何かを明言。
6. **これは単独信号の測定バグか、マルチ信号（複数信号同居）が根因か**を判定して報告
   （2信号が1窓に共存している記録である以上、task#6=マルチ信号の話と地続きのはず。切り分けを述べる）。

### (D) 修正方針の提案（提案のみ・実装しない）
7. 返り値の形（凍結シグネチャ）を保ったまま直せる**最小の修正案**を、案として文章で書く
   （例: detect_segments の連結条件にギャップ上限を設けて2信号を分離する／bw_median の median 対象を
   ノイズ床でなく信号ビンに限定する 等）。**複数案あれば trade-off も。実装は一切しない。**
8. 修正した場合に**既存の何が影響を受けるか**（classify の帯域幅依存ルール、既存テスト、
   過去記録の再測定）を洗い出す。
