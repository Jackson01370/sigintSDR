# 作業指示書: cnn-training M3 — classify への CNN 接続（監査役としての組み込み）

## 役割
あなたは sigscan プロジェクトの cnn-training エージェント。
本指示は、M1〜M2.5 で実データに通用するようになった CNN（方式軸・5クラス）を、
3段分類器（ルール → CNN → LLM）の **2段目として classify に正式接続** する作業である。
設計はユーザーと合意済み（下記「合意済み設計」）。実装はその設計の忠実な翻訳とする。

## 背景（なぜこの設計か）
- M2.5 で CNN は実データで BLE→narrowband-burst 93.9% / Zigbee 100% / WiFi→wideband-ofdm 53.8%。
- 残る WiFi の pulse-radar 予測 3 件を人間が画像+CubicSDR で確認した結果、**本物の WiFi**
  （周期的な顔をする WiFi＝ビーコン/ACK 列、および飽和級の混雑チャンネル）と確定。
  → 教訓: 「周期的な広帯域」は**見え方だけでは WiFi ともレーダとも言い切れない**。
    最終判定には**周波数の文脈**（2.4GHz ISM にレーダはまずいない）が要る。
  → よって CNN は「拒否権」ではなく **「監査役」**。方式の所見を出し、
    用途ラベルの最終判定は「ルール × CNN × バンドプラン文脈」の整合チェックが行う。

## 合意済み設計（これを実装する。逸脱禁止）

### 呼び出しタイミング
- **dwell（滞在観測）で保存候補になった信号の IQ** に対してのみ CNN を呼ぶ。
  サーベイ段階では呼ばない（重い・IQ が無い）。
- 入力は必ず凍結 `spec.render(iq, rate)` 経由（cnntrain.infer の既存経路
  `classify_iq` を再利用。新たな前処理を作らない）。

### 整合チェック（A/B/C）と確信度調整
ルール判定（用途ラベル＋確信度）と CNN 所見（方式クラス＋確信度）を、
期待対応表（用途→方式集合。probe の対応表を流用・一元化）で突き合わせる:

- **(A) 整合**: CNN クラス ∈ 期待集合
  → 確信度を **+0.10**（上限 0.95）。ラベル維持。
- **(B) 説明可能な不整合**: CNN クラス ∉ 期待集合 **だが**、バンドプラン文脈で
  説明がつく（初期実装は最小の文脈表で開始。少なくとも:
  「2.4GHz ISM 帯 × ルール=WiFi × CNN=pulse-radar → 周期的な顔の WiFi」を含む）
  → ラベル維持、確信度 **−0.05**（下限 0.05）。
- **(C) 説明不能な不整合**: 上記以外の不整合
  → 確信度を **min(rule_conf, 1 − cnn_conf)** に引き下げ（保守的）。
  → その結果 **0.7 未満なら用途 = Unknown（候補つき）** とし、既存の
    review.py フロー（人間判断）に乗せる。

係数（+0.10 / −0.05 / min式）は**最小実装の固定値**。根拠なきチューニング禁止
（妥当性検証は将来、データで行う）。

### 大原則（禁止事項の核）
1. **CNN の判定だけで用途ラベルを書き換えない**。CNN が強く反対しても、
   ラベルを差し替えるのではなく (C) により Unknown に落として人間に回す。
2. **既定 OFF**: CNN 接続はフラグ（例: `--cnn` または config の `cnn_enabled`、
   既定 False）でのみ有効。OFF 時の classify の出力は**従来と完全に同一**
   （既存テストを 1 件も変更せずに通ることがその証明）。
   チェックポイントパスは CLI/config で指定（例: `--cnn-checkpoint runs/m2_5/checkpoint.pt`）。
   フラグ ON かつチェックポイント不在は**明示エラー**（黙ってスキップしない）。
3. **凍結契約不可侵**: `classify.classify` のシグネチャ不変。spec.py / sigmf_io.py 不変。
   torch の import は CNN 有効時のみ遅延（OFF 運用や torch 無し環境で classify が
   壊れないこと）。
4. **マルチ信号同居は M3 では解かない**: 同居の可能性（例: CNN 確信度が低い・
   期待外）への特別対応は実装せず、CNN 所見をメタに記録するに留める（将来課題）。

### 来歴（メタ記録）
CNN を呼んだ記録には SigMF へ以下を残す（既存の extra_global / annotation
patch の流儀に従い、sigmf_io シグネチャ不変のまま呼び出し側で付与）:
- `sigscan:cnn_class`（方式クラス名）/ `sigscan:cnn_conf`
- `sigscan:cnn_verdict`（"A-consistent" / "B-context-explained" / "C-conflict"）
- `sigscan:cnn_checkpoint`（どのモデルか。ファイル名で可）
- 調整前後の確信度が追えること（例: `sigscan:rule_conf_pre`）。

2. `scheduler.py` の dwell→保存経路を読み、IQ と rate が classify 側へ渡せるか

   オプショナルなコンテキスト設定関数、または scheduler 側での後段監査——を検討し、
   採った方式と理由を報告。継ぎ目 classify.classify(引数) の変更は禁止）。
3. `cnntrain/infer.py` の `classify_iq` / `load_checkpoint` の入出力を確認。
4. probe の期待対応表の所在を確認し、**一箇所に一元化**（probe と classify が
   同じ表を import する。二重定義禁止）。
5. 既存テストのベースライン（124 passed, 3 skipped）を確認してから着手。

## 実装内容
1. **整合チェックモジュール**（例: `cnntrain/audit.py` または classify 内の小関数）:
   入力（rule_label, rule_conf, cnn_class, cnn_conf, band名/中心周波数）→
   出力（verdict A/B/C, 調整後 conf, 用途を Unknown に落とすか）。純関数にして
   テスト容易に。文脈表（B 用）はデータとして分離（拡張しやすく）。
2. **classify への接続**: CNN 有効時のみ、ルール判定後に infer.classify_iq を呼び、
   整合チェック→確信度調整→（0.7 未満なら Unknown 化）。LLM（3段目）の既存
   トリガ条件（低確信度）との関係を確認し、自然に直列になるよう接続
   （CNN で Unknown 化された場合に LLM が呼ばれる、等。既存挙動を壊さない）。
3. **scheduler/main の配線**: フラグ・チェックポイントパスの受け渡し。
   起動時表示（例: 「CNN分類器: 有効 (runs/m2_5)」）。SigMF への来歴記録。
4. **テスト（既存無変更・追加のみ・高速）**:
   - T12: 整合チェックの A/B/C と確信度調整が仕様どおり（純関数の単体テスト。
     A で +0.10 上限 0.95 / B で −0.05 / C で min 式と Unknown 化境界 0.7）。
   - T13: OFF 時の classify 出力が従来と同一（既存テストが無変更で通ることに加え、
     明示の回帰テストを 1 本）。
   - T14: ON 時の end-to-end（Sim で保存→メタに cnn_class/verdict が載る。
     チェックポイントはテスト内で極小学習 or M2.5 流用、実行時間に注意）。
   - T15: フラグ ON + チェックポイント不在 → 明示エラー。
5. **スモーク（実データ・推論のみ）**: 実 captures/ の数件に対し、scheduler を
   通さず classify 単体で CNN 監査を当て、(A)(B)(C) の分布を確認・報告
   （実データの書き換えは禁止。読み取りのみ）。
   期待: 今日確定した「2.4GHz の周期的 WiFi」が (B) で救われ、ラベル維持のまま
   確信度微減になること。

## やってはいけないこと（禁止事項の再掲）
- CNN の判定だけで用途ラベルを書き換えること（(C) は Unknown 化のみ）。
- classify.classify のシグネチャ変更。spec.py / sigmf_io.py の変更。
- 既定 ON にすること。OFF 時の挙動変更。既存テストの変更・削除。
- 期待対応表の二重定義（probe と別物を作ること）。
- 係数（+0.10/−0.05/min式/0.7境界）の根拠なき変更・チューニングループ。
- マルチ信号同居への特別対応の実装（記録のみ）。
- 実 captures/ の書き換え（スモークは読み取りのみ）。
- torch を classify の無条件 import にすること（OFF 運用を壊す）。
