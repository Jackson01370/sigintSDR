# 作業指示書: cnn-training M2 — simgen 実測アライン + 再学習 + 再プローブ + スポットライト初実装

## 役割
あなたは sigscan プロジェクトの cnn-training エージェント。
本指示は、M1.5 プローブと**人間による画像確認**で確定した「合成クラス定義と
実信号の見え方の乖離」を simgen に反映し、再学習 → 実データ再プローブで
改善を**数字で測る**作業である。あわせて、ユーザー発案の
「スポットライト機能」（イレギュラーな電波を目立つ場所に集める）を初実装する。

## 背景（画像確認で確定した事実 — これが設計の根拠）
M1.5 の不一致を人間が画像で検証した結果:
1. **BLE→cw-tone（49件）の真犯人は DC 残留線**。実 BLE は中心 2402.5MHz 付近の
   幅 1.5〜2MHz・時間の一部だけ点く**ブロック**（= 現行 narrowband-burst の定義に
   ほぼ合致）。一方、画像には**取得中心（0Hzオフセット）に全時間伸びる極細の
   薄い線（DC 残留）**が必ず写っており、合成データに DC 線が存在しないため、
   CNN は「教科書の cw-tone に一番似た特徴」= DC 線に反応していた。
   ※エージェント自身の M1.5 見立て（実BLE≈20kHz説）は、不自然と申し送り済みの
   bw_median 測定値に依拠しており、画像により**反証**された。
2. **WiFi→pulse-radar（残8件）は本物の WiFi**。13ms 窓の実 WiFi はパケット通信
   ゆえ「**不規則な間隔・可変幅の広帯域縦縞**」に見える。合成 wideband-ofdm を
   「全時間持続の塊」と定義したのが乖離の原因（= ドメインギャップ確定）。
3. **電子レンジ 1 件がルールラベル「WiFi」で混入**していた（大きなアーチ模様）。
   人間が確認し captures/_review_pending/ へ隔離済み → 現在の対象は **64 件**
   （BLE 49 / WiFi 13 / Zigbee 2）。プローブの不一致リストがこの誤ラベルを
   炙り出した = タイプ2 スポットライトの原型。
4. 13ms くじ引きは品質ゲート済みデータでは引き続き非顕在（noise-only 予測 0%）。

## 最重要原則（絶対厳守）
1. **凍結契約不可侵**: spec.py / sigmf_io.py 変更禁止。
   **DC 残留の注入は IQ（時間領域）合成で行い、凍結 spec.render を通す**。
   画像への後処理での線の描き込みは禁止（正準経路の純度を守る）。
2. **実データで学習・fine-tune しない**（推論のみ）。ただし、**ラベルを使わない
   物理量の計測**（実レコードの DC 残留強度を dsp.dc_spike_metrics 等で測り、
   注入パラメータの較正に使う）は**許可**する。教師信号は引き続き合成の真実のみ。
3. **captures/ の *.sigmf-data / *.sigmf-meta は読み取り専用**（移動・改名・上書き
   禁止）。スポットライトの PNG 出力先 captures/_images/_spotlight/ への書き込みは可。
4. **公平な before/after**: M1 と同一条件（per-class 60 / seed 0 / 8 epoch /
   同モデル）で再学習し、M1.5 ベースライン（下記に転記）と比較する。
5. バナー規律の継続: SYNTHETIC-ONLY（学習）/ REAL-DATA PROBE（探針）/
   **一致率は accuracy ではない**。
6. **チューニング無限ループ禁止**: 実装 → 学習 → プローブ → 報告で1サイクル。
   明白なバグ（例: DC 注入位置の誤り、sim val の崩壊）があれば修正して
   **1回だけ**再実行可。それ以上の精度狩りはしない（次サイクルの判断は人間）。

## 事前確認（実装前に必ず行い、結果を報告に含める）
1. captures/ 直下のレコード数 = **64**（BLE 49 / WiFi 13 / Zigbee 2）、
   電子レンジ 1 件（2448MHz_1780568765278_9）が _review_pending/ にあることを確認。
2. M1.5 ベースラインレポート runs/probe_real/probe_report.json の存在確認
   （比較に使う。無ければ本指示書の転記値を基準にする）。
3. **DC 残留の実測較正（ラベル不使用）**: 実レコード数件（静かな 2400MHz 帯中心）の
   IQ に対し dsp.dc_spike_metrics（または welch の中心ビン超過）で
   中心線の強度レンジを測り、注入パラメータの目標域を決めて報告。
4. 既存テストのベースライン（113 passed, 3 skipped）を確認してから着手。

## 実装内容

### (1) simgen の実測アライン（3本柱）
a. **DC 残留線の注入（全クラス共通・ランダム）**:
   - 全5クラスの IQ 合成に、確率 p（既定 0.7〜0.9 程度）で「中心 0Hz の微弱な
     DC 残留」を注入する。複素の微小オフセット + ゆっくりしたドリフト
     （remove_dc 後の残留を模す）。強度は事前確認3の実測レンジに合わせ、
     **クラスと無相関**（どのクラスにも同じ分布で入る = 識別の手がかりにならない）。
   - 検証: 注入後の IQ を spec.render し、中心列に薄い線が出ること、
     dc 超過が実測レンジ内であることをテストで確認。
b. **wideband-ofdm の再定義（実 WiFi バースト化）**:
   - 「全時間持続」をやめ、**不規則な間隔・可変長・可変デューティの広帯域
     バースト列**にする（12〜16MHz 幅は維持）。
   - **pulse-radar との分離を保つ**: pulse-radar は「厳密に周期的・短パルス・
     一様幅」、wideband-ofdm は「非周期・長短混在・帯域内の濃淡あり」。
     この対比が崩れていないか、生成パラメータの統計（間隔の変動係数等）で
     テスト確認。
c. **cw-tone のオフセット制約**:
   - cw-tone クラスの信号は**中心から |offセット| > 0.5MHz** に置く
     （注入 DC 線と重ならない）。「中心の細線 = アーティファクト、
     中心外の細線 = cw-tone」を学習可能にする。
   - narrowband-burst は現状の物理（幅 1〜2.5MHz・時間の一部点灯）を**維持**
     （画像確認で実 BLE と合致と判明したため）。noise-only も維持（DC 注入は入る）。

### (2) 再学習と sim 評価
- 新 simdata を生成（per-class 60, seed 0）→ runs/m2/ に学習（8 epoch, seed 0）。
- sim val の精度・混同行列を報告。**多少下がっても正直に報告**
  （クラスが現実的に難しくなるため。崩壊＝単一クラス潰れのみ要修正）。

### (3) 実データ再プローブ + before/after 比較
- M1.5 と同じプローブを captures/（64件）に実行 → runs/probe_real_m2/。
- **比較表**を作る。M1.5 ベースライン（転記。電子レンジ1件隔離前の値）:
  - BLE: 期待一致 0%（0/49・全件 cw-tone 予測）
  - WiFi: 期待一致 14.3%（2/14。pulse-radar 9 / cw-tone 3 / wideband-ofdm 2。
    うち pulse-radar 1 件は電子レンジと判明し隔離 → 今回は 13 件で比較し、
    その旨を注記）
  - Zigbee: 期待一致 0%（0/2・cw-tone 予測）
- 観測すべき方向（仮説）: BLE の cw-tone 予測が崩れて narrowband-burst が増える /
  WiFi が wideband-ofdm に寄る。**そうならなくても正直に数字を報告**。

### (4) スポットライト初実装（ユーザー発案）
- 目的: 「イレギュラーな電波を目立つ場所に集める」発見の漏斗。
- 出力先: **captures/_images/_spotlight/**（PNG + spotlight_report.txt）。
  元レコードは一切変更しない（PNG はコピー/新規レンダのみ。レンダは
  spec.render 経由で view_captures と同等の見た目）。
- 抽出基準（2タイプ・いずれも CLI 引数化）:
  - **タイプ1（自白型）**: ルールラベルが未識別/空、または rule confidence <
    閾値（**既定 0.5**。注意: 現収集物は conf≈0.62 が大量のため、0.7 を既定に
    すると洪水になる。この設計判断を報告に明記）。
  - **タイプ2（監査型）**: 期待対応表との不一致 かつ CNN confidence ≥ 閾値
    （既定 0.9）。確信度降順。電子レンジ発見がこの原型。
  - 両タイプとも **top-N で上限**（既定 15）し、洪水を防ぐ。
- spotlight_report.txt: 各エントリに（ファイル名 / タイプ / ルールラベル+conf /
  CNN 予測+conf / 理由の一言）。冒頭に「ここは発見の漏斗。隔離・ラベル修正の
  判断は人間が行う」と明記（自動隔離はしない）。
- 実装は probe のオプション（例: --spotlight）か小モジュール。**再利用可能な
  CLI** にする（今回限りのスクリプト禁止）。

### (5) テスト（既存無変更・追加のみ・高速）
- T5: DC 注入が中心に出る・強度が較正レンジ内・全クラスに無相関で入る。
- T6: cw-tone のオフセットが |>0.5MHz| を守る。
- T7: wideband-ofdm の非周期性 vs pulse-radar の周期性（生成統計の分離）。
- T8: スポットライト機構（合成データ代役で タイプ1/2 抽出・top-N 上限・
  レポート生成・元データ無変更）。
- 既存テストが**意図した物理変更**で落ちる場合は、勝手に書き換えず
  **中断して報告**（どのテストが何を固定していたかを添えて）。

## やってはいけないこと（禁止事項の再掲）
- spec.py / sigmf_io.py の変更。画像後処理での DC 線描き込み（IQ 合成で行う）。
- 実データでの学習・fine-tune（ラベル不使用の物理較正のみ可）。
- captures/ の *.sigmf-data / *.sigmf-meta の移動・改名・上書き・削除。
  スポットライトによる自動隔離（判断は人間）。
- classify.classify への接続・変更（M3 の仕事）。
- 既存テストの書き換え・削除（落ちたら中断・報告）。
- bw_median 等 capture-engine 側の測定量の「修正」（別件・スコープ外）。
- 一致率を accuracy と表現すること。チューニングの無限ループ。
